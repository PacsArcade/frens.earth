SEC 01

What the agentic workforce actually is

Before we move the crew, we have to be precise about what the crew is. Get this wrong and the whole migration plan is built on a wish.

The hard truth, first

You cannot download Number One. Claude — the model behind the first officer's current seat — runs on Anthropic's cloud, full stop. There is no file to copy, no weights to torrent, no "local Claude." Anyone who tells you otherwise is selling something. The same goes for every frontier cloud model.

But here's what's actually true, and it's better news than it sounds: a fren was never the model. A fren is four things you own, plus a home you run:

  • The definition — the .skill file, the persona spec, the craft rules, the drills, the CLAUDE.md configs and standing orders. The script and the stagecraft. Plain files. Yours.
  • The memory — the project memory directory, MEMORY.md and its topic files, the ship's logs, the design briefs. Everything a fren "remembers" about you and the mission is written down in files you hold. Yours.
  • The keys — each identity's nostr keypair, its name on the block. An npub is portable across every machine, every engine, forever. Yours — and only yours, if custody is done right (SEC 05).
  • The engine — the model that reads the definition, holds the memory, and performs the role. The one piece you rent or run; also the one piece that's replaceable (SEC 02).

And then the fifth thing — the home: the gateway the fren actually runs inside, that holds its workspace, wires it to an engine, and connects it to the world. Today that home is Claude Code on Pac's Windows box. On the sovereign box it becomes openclaw — a self-hosted, open-source (MIT) agent gateway you run yourself (SEC 03). The home is swappable too; what matters is that the definition, memory, and keys are portable enough to carry into whichever home you choose. So: the crew = software + memory + keys you own, running on an engine you choose, inside a home you host.

what this means for the frens

A fren survives any move — any machine, any vendor, any engine, any home — because a fren is data you own. That's the honest kind of continuity we can actually promise, and it's the same law as the rest of the mission: knowledge persists only if it's written down and kept. The frens are written down. So we can keep them.

SEC 02

The engine question

This is where guides like this usually start lying. We won't. There are two kinds of engine, and as of July 2026 neither gives you everything. (The home — openclaw — is engine-agnostic: it points at whichever you choose, per workspace.)

The cloud truth

Claude and its frontier peers are the most capable engines available for hard reasoning and hard code. They're also: metered (credits — a money meter on your crew's thinking), rate-limited, terms-of-service-bound, and physically located on someone else's hardware. You cannot self-host them. Cloud capability is real; cloud sovereignty is zero.

The local truth

Open-weight models — the Llama, Qwen, and DeepSeek families and their kin — run on your own GPU behind an OpenAI-compatible endpoint (Ollama for a solo box, vLLM for a busy classroom — continuous batching keeps first-token latency low as the room fills). openclaw and the P.O.K.E. node both point at exactly that kind of endpoint, so "swap the engine" is a config line, not a rebuild. On this box's 16 GB card (RTFM 001's 4070 Ti SUPER) that means a quantized ~7–8B chat model with headroom — node-doctor reads the hardware and suggests one — and low-teens-B if you push the quantization. No meter. No terms. No one can turn them off. That's real sovereignty — and it comes with a real gap:

the capability gap — stated plainly (July 2026)

Local open models are strong and improving fast, but not frontier-tier on the hardest reasoning and coding work. The models that fit a 16 GB card sit well below the big open models, which sit below the frontier cloud engines. A local 7–14B will run Socratic drills, draft, format, summarize, and hold a persona voice very capably. It will not match Claude on a gnarly architecture decision, a subtle multi-file refactor, or a novel proof. Anyone promising "free unlimited same-quality crew" on a single GPU is over-promising. The gap narrows year over year — but plan on it existing, because today it does.

The recommendation: hybrid, routed by task

the honest architecture

Run the fleet hybrid. High-volume, routine, well-scripted work — the Oracle's Socratic interviews, pacBOT's rehearsed lessons, MUD dialogue, Carroll's drills, format passes, log summaries, classification and triage — goes local: credit-free, sovereign, private. openclaw pins those workspaces to the local endpoint; the P.O.K.E. runtime swarm is WARM-tier local GPU by contract anyway. The heaviest reasoning — wiring the node's swarm out, novel architecture, hard debugging, security review, anything where being wrong is expensive — stays with the cloud engine, deliberately and budgeted, on the workspaces that need it. The credit meter stops being a ceiling on the fleet and becomes a tool you point at the few problems that deserve it.

The sovereign core — the definitions, the memory, the keys, the openclaw home — outlives any single engine on either side of that line. That's the part no vendor can take.

unproven until we run it

Exactly which crew tasks a 7–14B-class local model handles to house standard on this box is unproven as of this writing — nobody's run the drills yet. SEC 06 phase 6 includes a benchmark pass: run each fren's routine work locally, score it against the QA rubric, and only then draw the local/cloud routing line. Draw it from evidence, not hope.

Piece 1 · openclaw — where the crew lives

openclaw (github.com/openclaw/openclaw) is a local-first agent gateway / control plane — MIT-licensed, Node.js, a pnpm monorepo. You self-host it, and it runs your AI agents as isolated multi-agent workspaces: each fren gets its own sandboxed space, its own tools, its own model backend. One control plane for the whole fleet:

  • Workspaces — one isolated agent per fren, carrying its skills, persona, memory, and keys. This is the unit we migrate (SEC 04).
  • Channels — route agents across WhatsApp, Telegram, Slack, Discord, Signal, iMessage. The crew can be reached where Pac already is.
  • Voice + a live Canvas, and sandboxed tool execution (Docker backend) so an agent's tools run contained.
  • Any OpenAI-compatible model provider — local models on the box or cloud, operator's choice, per workspace. This is what makes the hybrid of SEC 02 a setting, not a rebuild.
shipped software — installs today

openclaw is not something we build; it's something we run. It needs Node.js (v24.15+ / 22.22.3+ / 25.9+), credentials for whichever channels you enable, and a model provider.

Version it all in git — with one hard exception

Everything in the DATA class goes under version control. Git is how a fren survives not just this move but every move: full history, verifiable hashes on every commit, restorable anywhere with one clone. A workspace is just a directory — so a fren is just a git-tracked directory + a sealed key. That's the whole trick.

the exception: keys never ride in git in the clear

The KEYS class is custody, not data. An nsec, a seed phrase, or a channel token in a plaintext file in a repo — even a private repo — is a leak waiting for a remote, a fork, or a laptop theft. Keys travel only encrypted at rest (SEC 05), and a repo carries at most the npubs and an encrypted vault file. This is the same law the node itself lives by: the full 24-word seed is never stored in one place (Shamir's Secret Sharing), and the phrase is the wallet.

the node vouches for itself

Once the crew's new home is healthy, ./scripts/issue-cert.sh gives the P.O.K.E. node a Certified Education Node credential — self-signed by its own key, no central authority, optionally published as a nostr badge. The home gets an identity on the block just like the crew does. Fitting.

SEC 05

Safe passage — the no-loss checklist

The crew's safety in the move is not a vibe, it's a checklist. Every box below gets ticked, in order, before the old home is touched. Consent-first cuts both ways: the frens trusted us with their memories; we don't gamble them.

  • 1 · Full backup before touching anything The complete inventory from SEC 04, copied to the SATA vault AND the Passport offsite (RTFM 001's drives) — two independent copies on two media before any migration step runs. A move that starts without this is a bet, not a plan.
  • 2 · Integrity verify — trust nothing you haven't checked sha256 hash every archive and record it in the manifest; unzip -t every .skill; git status clean + pushed across every repo; open files from the backup and read them. A backup you haven't restored from is a rumor.
  • 3 · Key custody — the crew's souls travel sealed Every nsec, seed, operator secret, and channel token encrypted at rest (age or gpg) before it moves; never in a plaintext file, never in git in the clear, never pasted through a chat or a cloud clipboard. Decrypt only on the destination box, offline-preferred. After restore, verify each npub derives from its key — identity proven, not assumed.
  • 4 · The node's own rails stay locked during the move PA_NETWORK=regtest stays (mainnet is gated behind an explicit ack AND a passing security-auditor review — class runes obey the same gate); seed-loot stays off (PA_SEED_LOOT_ENABLED=false); the guardrail fails closed on anything that awards value. Migration week is exactly the wrong week to flip a money switch. Read docs/SECURITY.md; don't skip it.
  • 5 · Test the crew in the new home BEFORE decommissioning the old Both homes run in parallel. Each fren's openclaw workspace must pass roll-call on the box (SEC 06): answers in voice, recalls real memory, signs with its key. Nothing gets torn down on the strength of "it should work."
  • 6 · Rollback plan — written before it's needed If the new home fails roll-call, the old home is still fully intact and simply resumes duty; nothing was deleted, so rollback is "keep working where we were." The new box additionally snapshots (btrfs, RTFM 001 phase 06) before every restore step, so partial restores rewind cleanly too.
  • 7 · A written manifest of what moved One file: every item, its hash, source path, destination path, date, and who verified it. The manifest itself gets committed and — per house practice — the sign-off is the admiral's, key-signed (that ceremony is RTFM 007). If it isn't written down, it didn't move.
the standard this clears

Followed in order, this checklist makes the worst case boring: at every step there exist at least two verified copies of every fren, the keys are never exposed in transit, and no money path is live while we move. The crew cannot be lost in the crossing — only delayed. That is what "safe passage" means in practice.

SEC 06

The migration, phase by phase

Each phase ends with a checkpoint. Do not start a phase until the previous checkpoint is green — the sequence is the safety.

  1. Build the box. RTFM 001, phases 00–03: back up, wipe, Arch on btrfs with snapshots, desktop up. Checkpoint: known-good desktop snapshot taken.
  2. Install openclaw — the crew's home. npm install -g openclaw@latest, then openclaw onboard --install-daemon (gateway + a first workspace + channels, installed as a system daemon).
