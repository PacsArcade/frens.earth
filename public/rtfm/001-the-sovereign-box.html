PHASE 00

Back up & escape Windows

The only irreversible part of this whole build is the wipe. Everything here exists so that when you erase Windows, nothing you care about goes with it.

Your disks have a plan

You have four internal drives. Three are effectively empty — which means you can carve out a real backup vault and a games library without buying anything.

What's actually irreplaceable

Most of C:\dev is already in git/on GitHub — that's not what you're saving. Hunt for the things that exist only here:

  • Keys & seeds — the Akron space-owner wallet recovery phrase (the phrase is the wallet — confirm it's written offline, never a plaintext file), any .env.local / BLOB_READ_WRITE_TOKEN / operator keys, SSH keys (%USERPROFILE%\.ssh), gh tokens.
  • The wallet source set — C:\dev\Reference\wallet\ (the 14 official Ashigaru zips) and C:\dev\research.
  • Uncommitted work — anything under C:\dev not pushed (check git status across repos), plus C:\dev\*.md notes (pet-game-design, PASSDOWN…).
  • Personal — photos, documents, browser profiles/bookmarks, anything in Desktop / Documents / Downloads you'd miss.
Verify, then — and only then — wipe

Copy the irreplaceable set to the SATA vault, and put the critical-few (keys, seeds, wallet zips, docs) also on the external Passport. Then open several files from the backup and, for the important archives, check a hash. A backup you haven't restored-from is a rumor.

 powershell — from Windows, before wiping
# find every git repo with unpushed / dirty state under C:\dev
PS> Get-ChildItem C:\dev -Recurse -Directory -Filter .git -Depth 3 |
      ForEach-Object { $r=$_.Parent.FullName; git -C $r status -s |
      Where-Object { $_ } | ForEach-Object { "$r  $_" } }

# mirror a folder to the SATA vault (robocopy = resumable, verifiable)
PS> robocopy "C:\dev\Reference" "E:\vault\Reference" /MIR /R:2 /W:5 /XA:SH /NP
Point of no return

Windows 11's key is digital-entitlement bound to this board, so a reinstall is always possible later — but you said Arch forever, so this is a clean break. Do not proceed past Phase 01 until the vault is verified and the Passport copy exists. Unplug the vault + Passport during install so they can't be selected by accident.

PHASE 01

BIOS & the install USB

A few firmware settings now save hours later — memory speed, a clean UEFI boot, and getting the NVIDIA box to hand the installer a working screen.

In the ASUS BIOS (Del at boot)

  • EXPO I — turn it on. Your Corsair kit is rated DDR5-6400 but idling at 4800; EXPO takes it to its rated speed (on AM5, 6000 is the stable sweet spot if 6400 won't post).
  • Resizable BAR — on (helps the 4070 Ti Super).
  • Secure Boot — set to Other OS / disabled for the install. (You can re-enable it later with signed images via sbctl — optional, noted in Phase 06.)
  • Boot — UEFI only, CSM off. Fast Boot off while installing.
  • Primary display — leave PCIe/Auto; the dual-GPU decision happens in Phase 03.

Make the boot media

Grab the latest ISO from archlinux.org/download (verify the signature). Fleet method: one Ventoy stick — install Ventoy to the USB once, then just copy ISO files onto it (Arch + Clonezilla ride the same stick; see RTFM 011 Phase 0). Classic alternatives: Rufus in DD image mode from Windows, or dd from any Linux.

 write the ISO (from a Linux box)
$ # find the stick — be certain of the device node!
$ lsblk
$ sudo dd if=archlinux-x86_64.iso of=/dev/sdX bs=4M status=progress oflag=sync
Boot it

Spam F8 at post for the ASUS boot menu, pick the USB (UEFI entry). You'll land at a root shell. If Wi-Fi is your only network, connect with iwctl (station wlan0 connect <SSID>) — the installer kernel already has the driver for the board's Wi-Fi.

PHASE 02

Install Arch — btrfs, snapshots, GRUB

We install onto the SN850X only, on btrfs so the whole system can be snapshotted and rolled back — which is what makes "Arch forever" safe instead of scary.

During the crossing: follow 011

The live command-by-command runbook is RTFM 011 — The Install, kept in lock-step with this page (Ventoy stick, btrfs + GRUB, KDE, the MediaTek Wi-Fi firmware). This page stays the why and the post-install life below (NVIDIA, gaming, sovereign stack, snapshots). If the two ever disagree, 011 is the runbook of record for the install and this page is the reference — flag the drift to the crew.

archinstall vs by hand

The guided archinstall (ships on the ISO) is genuinely good now and can do btrfs + subvolumes + systemd-boot in one pass — perfect for a first build. The manual route below gives you the exact subvolume layout snapshots love. Either is fine; the manual key steps:

 partition + btrfs subvolumes on the SN850X
# identify the 4TB NVMe (was Windows). Say it's /dev/nvme0n1
# lsblk -d -o NAME,SIZE,MODEL
# EFI (1G) + root (rest). GPT.
# mkfs.fat -F32 /dev/nvme0n1p1
# mkfs.btrfs -f -L arch /dev/nvme0n1p2

# # subvolumes: @ = root, @home, and split-outs snapshots skip
# mount /dev/nvme0n1p2 /mnt
# for v in @ @home @snapshots @var_log @pkg; do btrfs su cr /mnt/$v; done
# umount /mnt
# mount -o noatime,compress=zstd:1,subvol=@ /dev/nvme0n1p2 /mnt
# mkdir -p /mnt/{home,.snapshots,var/log,var/cache/pacman/pkg,boot}
# # mount @home, @snapshots, @var_log, @pkg to those, EFI to /boot
 base system
# pacstrap -K /mnt base linux linux-firmware linux-headers \
      btrfs-progs amd-ucode \
      networkmanager sudo vim git base-devel zsh
# genfstab -U /mnt >> /mnt/etc/fstab
# arch-chroot /mnt
# timezone, locale, hostname, root pw, a user in wheel, enable NetworkManager
# systemctl enable NetworkManager
# useradd -mG wheel -s /usr/bin/zsh pac  &&  passwd pac
Board note — Wi-Fi firmware

The B650E-F's Wi-Fi/BT is a MediaTek MT7922 (driver mt7921e). Arch split linux-firmware into vendor subpackages in 2025, so explicitly install linux-firmware-mediatek or Wi-Fi is dead on first boot. If Wi-Fi runs slow later: options mt7921e disable_aspm=1 in /etc/modprobe.d/.

Bootloader — GRUB + grub-btrfs (one-click rollback)

systemd-boot is simpler, but on an NVIDIA daily-driver the safety net wins: grub-btrfs turns every btrfs snapshot into a bootable menu entry, so a bad driver update is "reboot → pick yesterday's snapshot", not a rescue-USB night. Install grub + grub-btrfs + efibootmgr, generate the config, and your snapshots appear in the boot menu.

Checkpoint

Reboot, remove the USB, and land on a login shell on Arch with networking. That's the mountain climbed — everything after is additive and reversible via snapshots.

PHASE 03

NVIDIA + desktop — the dual-GPU decision

The one genuinely fiddly bit on this box: you have an NVIDIA 4070 Ti Super and the 7800X3D's Radeon iGPU. Getting that right up front avoids a black screen.

Recommended: leave the iGPU on, plug into the 4070 Ti Super

This is a desktop, not a laptop — no PRIME/Optimus, no offload config. Plug the monitor into the NVIDIA card and it's the one render GPU. Keep the Radeon iGPU enabled: it costs nothing, gives you a recovery display output if an NVIDIA update ever breaks the desktop, and keeps the door open for a GPU-passthrough VM later (see RedM, Phase 04). Bonus: the "disable iGPU" BIOS toggle is reported flaky on some ASUS AM5 boards, so leaving it on is the lower-risk default.

 driver — Ada uses the OPEN module (Arch default since driver 590)
$ # nvidia-open is the default for Turing+ (your Ada card). Plain build
$ # for the stock `linux` kernel; -dkms only if you run custom kernels.
$ sudo pacman -S nvidia-open nvidia-utils lib32-nvidia-utils egl-wayland
$ # add modules to /etc/mkinitcpio.conf, then rebuild:
      MODULES=(nvidia nvidia_modeset nvidia_uvm nvidia_drm)
$ sudo mkinitcpio -P
$ # nvidia_drm.modeset/fbdev are ON BY DEFAULT now (nvidia-utils 560.35+),
$ # so no kernel params needed — Wayland just works on the 590 branch.

Desktop — KDE Plasma (Wayland)

For an NVIDIA gaming box, KDE Plasma is the pragmatic pick: best-in-class on Wayland now, HDR + VRR + per-display refresh, and a sane path back to X11 if a game needs it.

 plasma + audio
$ sudo pacman -S plasma-meta sddm konsole dolphin \
      pipewire pipewire-pulse pipewire-alsa wireplumber
$ sudo systemctl enable sddm
Checkpoint

Reboot into Plasma. nvidia-smi shows the 4070 Ti Super; glxinfo | grep renderer and a Plasma "System Information" both name NVIDIA. Take your first snapshot here (Phase 06) — this is "known-good desktop".

PHASE 04

Gaming — RDR2, Steam & a RedM world

This rig is built for it: the 7800X3D is the best gaming CPU made, and 16GB of Ada handles RDR2 maxed. Two jobs here — play, and host the RedM world.

The gaming base

 enable multilib (/etc/pacman.conf) then install
$ # uncomment [multilib] in /etc/pacman.conf, then:
$ sudo pacman -Sy steam gamemode lib32-gamemode mangohud lib32-mangohud \
      lutris protontricks vulkan-icd-loader lib32-vulkan-icd-loader
$ # from the AUR: protonup-qt (installs Proton-GE), goverlay (mangohud GUI)

Red Dead Redemption 2 — a solved case

Single-player RDR2 runs great here (ProtonDB Gold, no anti-cheat). The recipe:

  • Steam → RDR2 → Properties → Compatibility → Force Proton Experimental (most stable; GE-Proton 9-27+ also fine, but skip GE 10-31/10-32 — known regression — fall back to 10-30 if a new GE won't open the launcher).
  • The real gate is the bundled Rockstar Games Launcher: first run needs a live login inside the Proton prefix, then it caches and speeds up. Social Club error? add WINEDLLOVERRIDES=vulkan-1=n,b %command% to launch options.
  • Renderer: since VKD3D-Proton 3.0 (Nov 2025) the in-game DX12 path runs clean on Linux — and the in-game Vulkan renderer stays the no-drama fallback.

RedM — host the world (yes), play the client (not natively)

The two halves of RedM split hard across Linux — and the client half is exactly the wall you hit on Fedora:

Playing RedM natively: a hard no (confirmed May 2026)

The RedM/FiveM client won't run under Proton/Wine — Cfx.re's "Adhesive" anti-cheat has no Linux build and fails to initialize on every Proton version. That's an upstream lock, not a distro problem, so Arch won't fix it any more than Fedora did. Don't sink another month into Wine for the public client.

Your own world changes the math — the sovereign play

Adhesive is only enforced on Cfx's public servers. On a server you own, you can turn it off — which is the one context where a custom, Linux-friendly interface is even on the table. That's the direction you're pointing at: don't fight Cfx's anti-cheat on their turf — bring RedM (then the next game, then the next) into worlds you host on the P.O.K.E. network, gated by SCAR. Own the stack, own the client, own the rules. (For public servers meanwhile: a Windows GPU-passthrough VM using the dGPU — which is why we kept the iGPU on — or a dual-boot sliver.)

 host the world — FXServer + txAdmin (Linux-native)
$ # latest Recommended build (proot Linux artifact)
$ wget https://runtime.fivem.net/artifacts/fivem/build_proot_linux/master/<latest>/fx.tar.xz
$ mkdir -p ~/RedMServer && tar xf fx.tar.xz -C ~/RedMServer
$ cd ~/RedMServer && ./run.sh   # txAdmin setup URL + PIN on :40120
$ # game port 30120 tcp+udp · needs a Keymaster key from cfx.re
$ # wrap in a systemd unit (Restart=on-failure) so the world stays up
PHASE 05

The sovereign stack

The reason the box exists. These run as systemd services so they come back on every boot — the companion daemons that frens.earth (the web layer) talks to.

Bitcoin Core — the foundation & the heartbeat

Everything anchors here: the Spaces node rides on it, and BFT (the "Bitcoin heartbeat") reads it. Mainnet + txindex is ~750 GB and growing — put the datadir on the FIKWOT 2TB (or the SATA vault), not the OS drive.

 bitcoind as a service
$ sudo pacman -S bitcoin-daemon   # or build/download core; verify sigs
$ # /etc/bitcoin/bitcoin.conf: txindex=1, datadir=/mnt/data/bitcoin,
$ # server=1, rpccookiefile — mirrors your bitcoin-node-deploy config
$ sudo systemctl enable --now bitcoind

Spaces — spaced + space-cli

Built from source (Rust). spaced scans Bitcoin for the protocol and serves the JSON-RPC on :7225 that the frens.earth admin console connects to; space-cli holds the @frens / @pacsarcade wallet. Keys live here, on the box — never on Vercel.

 build + run spaced
$ sudo pacman -S rustup && rustup toolchain install stable
$ git clone https://github.com/spacesprotocol/spaces && cd spaces
$ cargo build --release   # → spaced, space-cli
$ # systemd unit: spaced --chain mainnet --bitcoin-rpc-cookie …
$ # then: space-cli wallet import  (the recovered Akron phrase, over SSH only)

The MUD engine + LLMs

The knowledge-engine MUD (Python) runs here too — the persistent world, the admin/ticketing API on :4001, telnet/WS surfaces, and the LLM hookups. Local models fit this GPU: Ollama on 16GB of VRAM comfortably runs quantized 7–14B models (and 32B at tight quants) for the arcade's own brain, with cloud APIs alongside.

 python service + local LLM
$ sudo pacman -S python uv ollama-cuda   # ollama-cuda is in official extra now
$ sudo systemctl enable --now ollama
$ ollama pull qwen2.5:14b   # 14B @ Q4_K_M fits 16GB with headroom
$ # room for Q5_K_M, or ~27B at IQ3 if you push it; force full GPU offload
$ # knowledge-engine MUD → its own systemd unit; admin API on :4001
How it ties together

frens.earth (on Vercel or served from the box) is the web layer; bitcoind + spaced + the MUD engine are the companion daemons on this box. The admin console's "connect your node" panels point at 127.0.0.1:7225 (spaced) and 127.0.0.1:4001 (MUD) — local, behind your own walls.

PHASE 06

Forever — snapshots & upkeep

"Arch forever" works because you can always roll back. Set this up once and a bad update is a two-minute fix, never a reinstall.

  • snapper + btrfs-assistant — automatic snapshots of @ and @home; a pacman hook (snap-pac) snapshots before every upgrade so you can boot the previous state.
  • Boot into a snapshot — grub-btrfs or a systemd-boot snapshot entry gives you "roll back the whole OS" from the boot menu.
  • The vault stays cold — periodic btrfs send/receive (or restic) from the box to the SATA vault; the critical-few keep mirroring to the Passport for offsite.
  • Fans & sensors — lm_sensors + sensors-detect; this board's Nuvoton chip (nct6775) is in-kernel, but ASUS Q-Fan can hide the PWM interface. If fans don't show, set the headers to Auto-Detect / disable Q-Fan in BIOS, then fancontrol. A "works with a little fiddling" item, not plug-and-play.
  • Update rhythm — pacman -Syu deliberately, read the Arch news first; snapshots make it low-stakes.
  • Secure Boot (optional) — sbctl to sign your kernel + NVIDIA modules and turn it back on, if you want it.
The finish line

A box that games at the top of the stack and keeps @frens alive — with an undo button on the whole operating system. Knowledge kept, tied to the block, ours.
