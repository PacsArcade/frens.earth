RTFM · 006 · deploy from SCAR with your signature

Ship it.
Merge ≠ live until you say so.

frens.earth deploys on your command, not on a git push. This wires a signed ▲ SHIP button into SCAR so you deploy production with your key — no terminal — while a push to main never deploys on its own.

Why this exists. Merging records the code; shipping puts it in front of the world. Those are two different acts, and both should be your signature. This keeps them separate — and both yours.
01

Connect the repo (once)

Deploy Hooks — the thing the SHIP button fires — need the project linked to its Git repo. This does not mean pushes will auto-deploy; step 02 turns that off.

  • Vercel dashboard → the frens-earth project → Settings → Git.
  • Connect Git Repository → GitHub → PacsArcade/frens.earth.
Note

Every deploy so far has been a CLI upload (vercel deploy). Connecting the repo just unlocks Deploy Hooks — it changes nothing else until you push, and step 02 stops pushes from deploying.

02

Turn off auto-deploy

By default a connected repo deploys on every push to the production branch. We don't want that — a merge should sit in main until you ship it. Flip it off in vercel.json at the repo root:

 vercel.json — add the git block
{
  "git": {
    "deploymentEnabled": false   // no branch auto-deploys on push
  }
}

Commit it. From now on, pushing to main creates no deployment — Vercel just records the commit.

The key fact

deploymentEnabled:false blocks automatic (git-push) deploys only. Deploy Hooks still fire — they're an explicit trigger. That's exactly the split we want: push does nothing, the button ships.

Per-branch variant

If you'd rather only stop production: "deploymentEnabled": { "main": false }. For frens.earth (all deploys are hook/CLI), false for everything is simplest.

03

Create the Deploy Hook

A Deploy Hook is a secret URL. POST to it → Vercel builds and deploys the branch you bound it to. Make one for main.

Dashboard

  • Settings → Git → Deploy Hooks.
  • Name it ship-main, branch main → Create Hook → copy the URL.

…or the CLI (returns the URL)

 make the hook from the terminal
$ vercel deploy-hooks create ship-main --ref main
# → https://api.vercel.com/v1/integrations/deploy/prj_…/…
Careful — it's a key

Anyone with that URL can trigger a production deploy. Treat it like a token: don't paste it in chat, don't commit it. It goes into SCAR write-only (step 04) and is never shown again.

04

Arm the SHIP button

  • Open SCAR → ACTION ITEMS (/a).
  • In the SHIP panel's connect box, paste the Deploy Hook URL → save. It's stored write-only (masked forever after, like your GitHub token).

Done. The panel now shows the ▲ SHIP — DEPLOY TO PRODUCTION button.

05

The flow, from now on

  • Sign to merge — authorize a PR in SCAR (your key merges it).
  • Sign to SHIP — press ▲ SHIP; you sign PACS-DEPLOY-<ts>, the server verifies your key against the operator allowlist, fires the hook, and records the ship (your key · block height · Vercel job id).
  • Live. No terminal, ever again.
Checkpoint — did it work?

Push a throwaway commit to main → Vercel shows no new deployment. Press ▲ SHIP in SCAR → a fresh Production deployment appears, and a row lands in the SHIP log, block-stamped and signed by your key.
