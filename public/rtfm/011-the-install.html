When a name/size is unique to your box, I say so — read your own screen. The why behind every choice here lives in RTFM 001 — this page is the runbook; they are kept in lock-step.

PHASE 0 · Windows — the Ventoy stick ✅ done

First move: pause Windows Update. Settings → Windows Update → Pause for 5 weeks. A forced update already rebooted the bridge mid-watch and ate a set of review notes; it must NOT reboot the box mid-backup or mid-image. (The crew's blackbox rule in REPORT IN below is the other half of "never again".)

One stick, many ISOs — Ventoy (ventoy.net) turns a USB stick into a boot menu you just drag ISO files onto. No re-flashing per ISO, ever.

  1. Download Ventoy on Windows → run Ventoy2Disk.exe → select the stick → Install (this erases the stick, once).
  2. Copy the ISO files straight onto the stick: Clonezilla (clonezilla.org — "alternative stable", amd64) and Arch (archlinux.org/download) — current mission stick carries archlinux-2026.07.01-x86_64.iso.
  3. That's it. At boot, Ventoy shows a menu of every ISO on the stick — pick the one the phase calls for.

PHASE 1 · Windows — plain-file backup (the easy-restore copy) ✅ done — verified on the SATA vault

A Clonezilla image is a sealed full-disk snapshot — hard to pull single files from. So also copy the folders that don't travel with git as normal files. Plug in the 4 TB backup drive (assume it's E: — check in File Explorer). Open PowerShell:

robocopy C:\dev E:\backup\dev /E /R:1 /W:1
robocopy C:\Users\pacman\.claude E:\backup\claude /E /R:1 /W:1
This is the soul of the ship. C:\Users\pacman\.claude is the crew's memory — it is NOT in any repo. If it's not on the backup drive, it's gone forever after Phase 3. Open E:\backup\claude and confirm files are really there before moving on.

Copy anything else you care about the same way (e.g. C:\dev\studies is already inside C:\dev, so it's covered).

PHASE 2 · Clonezilla — full-disk safety net ✅ done

  1. Leave the backup drive plugged in. Restart. At the maker's logo, tap the boot-menu key (F8 on this ASUS board). Pick the Ventoy USB (UEFI entry) → in the Ventoy menu pick the Clonezilla ISO.
  2. Choose Clonezilla live (default) → language → keep keymap → Start Clonezilla.
  3. device-image → local_dev → press Enter to let it scan → pick the backup drive as where the image is saved (by size/label) → / (top dir).
  4. savedisk → name it e.g. windows-2026 → select the source disk = the NVMe that has Windows (by size + model).
  5. Accept defaults (check source, no encryption) → y → y. Let it run (can take a while). At the end: poweroff.
✅ Now you have BOTH: a browsable file copy (Phase 1) and a full-disk image (Phase 2) on the backup drive. Windows is safe. You can restore the whole thing later if you ever want to.

PHASE 3 · Boot the Arch installer + get online

BIOS preflight first (from RTFM 001 — Del at boot into the ASUS BIOS): Secure Boot → "Other OS" / disabled (Ventoy + the Arch ISO won't boot signed), CSM off (UEFI only), Fast Boot off, EXPO I on (DDR5-6400 rated; if the box won't POST or the installer crashes, drop EXPO to 6000 or off — memory stability first, speed later). Save & exit.

  1. F8 at the logo → pick the Ventoy USB (UEFI entry) → in the Ventoy menu pick archlinux-2026.07.01-x86_64.iso → if a second menu asks, Boot in normal mode → "Arch Linux install medium". You land at a black root@archiso ~# prompt.
  2. WiFi:
    iwctl
device list
    If you see a device like wlan0, continue:
    station wlan0 scan
station wlan0 get-networks
station wlan0 connect "YOUR_WIFI_NAME"
# type your wifi password when asked
exit
ping -c3 archlinux.org
    Replies = you're online.
  3. If device list is EMPTY, your WiFi card's driver isn't in the installer (common on brand-new cards). Don't fight it — plug an Ethernet cable from your router into the board. It connects automatically. Test: ping -c3 archlinux.org. (You'll set WiFi up properly after install, in Phase 5.)

PHASE 4 · Install Arch — the guided way

Before you type anything: unplug what must survive. Power down if needed and disconnect the 4 TB SATA vault (ST4000DM004) and any USB drive. A drive that isn't connected cannot be wiped by a mis-pick. The target is the WD_BLACK SN850X 4 TB NVMe (the one that had Windows) — this box has more than one NVMe, so identify by model + size, never by position in the list. To double-check from the prompt first: lsblk -d -o NAME,SIZE,MODEL.
btrfs vs ext4, in arcade terms. ext4 is the classic cartridge: solid, simple, fast — but no save states. btrfs does the same job plus save states for the whole operating system: it snapshots itself before every update (snap-pac), and GRUB can boot straight into yesterday's save if today's patch bricks the desktop — a bad update becomes a reboot, never a reinstall. Cost: a hair of speed and more machinery under the hood. On a daily-driver ship you never want to rebuild, the save states win. That's the whole difference as it matters here.

Type:

archinstall

A menu opens (arrow keys, Enter to open, Esc/Tab to go back). Menu wording shifts between ISO versions — match by meaning, not exact words. Set these:

  1. Mirror region → your country.
  2. Disk configuration → "Partitioning" → Use a best-effort default partition layout → pick the WD_BLACK SN850X by model + size → filesystem btrfs → say yes to the default subvolume structure and yes to compression → if it offers snapshots, pick Snapper. Confirm it will wipe that disk. ← this erases the NVMe. Model must read SN850X — not ST4000 (vault), not FIKWOT, not SN770. (Why btrfs: RTFM 001's "Arch forever" plan — the whole OS snapshots and rolls back, so a bad update is a reboot, not a reinstall.)
  3. Bootloader → GRUB. (Why not systemd-boot: grub-btrfs puts every snapshot in the boot menu — the undo button for the whole OS. This supersedes the earlier draft of this page that said systemd-boot.)
  4. Hostname → e.g. frenship.
  5. Root password → set one you'll remember. Chosen in your head — never typed on the laptop.
  6. User account → Add a user (e.g. pac), give it a password, answer yes to "superuser (sudo)".
  7. Profile → Desktop → KDE Plasma (friendly, looks like Windows). [Choose Minimal only if you want no GUI.]
  8. Graphics driver (asked with the profile) → Nvidia (open kernel module) if listed — right for the 4070 Ti Super (Ada); otherwise Nvidia (proprietary). Either boots; 001 Phase 03 refines.
  9. Audio → pipewire.
  10. Network configuration → Use NetworkManager. ← don't skip; this is what makes WiFi work after reboot.
  11. Additional packages → type: git base-devel nodejs npm nano linux-firmware-mediatek (the last one is the board's MT7922 WiFi firmware — Arch split linux-firmware into vendor packages in 2025; without it WiFi can be dead on first boot. See 001.)
  12. Optional repositories → enable multilib (Steam and the gaming stack in 001 Phase 04 need it).
  13. Timezone → yours.
  14. Scroll to Install → Enter → confirm. Wait. When it asks about chroot, say no. Then Reboot → pull the USB out as it restarts.
Reconnect the SATA vault (and nothing else yet) after the first successful boot.

PHASE 5 · Arch — first boot

  1. Log in as your user (pac).
  2. WiFi: click the network icon (bottom-right) → your network → password. (Or run nmtui in a terminal.)
  3. Open a terminal (Konsole) and update everything:
    sudo pacman -Syu
  4. If you installed NVIDIA, reboot once more: reboot.
  5. Arm the undo button (the reason we chose btrfs + GRUB):
    snapper list-configs   # if archinstall set up Snapper, "root" is listed — done
sudo pacman -S snapper snap-pac grub-btrfs inotify-tools
sudo systemctl enable --now grub-btrfsd
    If list-configs came back empty, the Snapper config needs the Arch-wiki dance — do it with the crew once they're home (Phase 7), not solo. snap-pac snapshots before every pacman upgrade; grub-btrfs puts snapshots in the boot menu. Full upkeep story: 001 Phase 06.

PHASE 7 · Arch — bring the crew + data home

  1. Plug in the 4 TB backup drive. Find it:

    Look for your backup drive by size/model (ST4000DM004); note its partition (e.g. sdb1).

Your memory (~/.claude), your studies (the pupil clock etc., under ~/dev), and the repos are now all on the new box.

RESCUE MISSION · Laptop → new ship — the Degen Wonderland originals

On the laptop live hand-drawn images from the actual artists for Degen Wonderland. They are irreplaceable — treat them like keys: never one copy, verify every hop, and the laptop is not wiped until the numbers say every file made it. They're image files (data, not programs) so carrying them off the infected shuttle is safe — just never run anything from that machine.

Timing (the admiral's call): this runs after the new ship has proven stable — no rush. The copying is the admiral's own hands; Steps 1–2 are there as the map. Steps 3–5 are the contract: whoever carries the cargo, the verifications run, and the wipe stays gated on them.

Step 1 — laptop: find every original. Sweep beyond the obvious folder — Desktop, Downloads, Pictures, scanner output. This lists the 25 folders holding the most big images:

Get-ChildItem C:\Users -Recurse -File -ErrorAction SilentlyContinue `
  -Include *.png,*.jpg,*.jpeg,*.tif,*.tiff,*.psd,*.bmp,*.heic,*.svg,*.pdf,*.procreate |
  Where-Object Length -gt 500KB | Group-Object DirectoryName |
  Sort-Object Count -Descending | Select-Object -First 25 Count,Name

Step 2 — laptop: copy to the USB hard drive with a log. (Assume the USB drive is F: — check File Explorer. Swap C:\path\to\DW-ART for the real folder; repeat per folder if the art lives in several.)

robocopy "C:\path\to\DW-ART" "F:\rescue\dw-art" /E /COPY:DAT /R:2 /W:5 `
  /LOG:F:\rescue\robocopy-dw-art.log /TEE

Step 3 — laptop: verify the copy, byte for byte. Equal counts + empty Compare-Object output = every file's SHA-256 exists on both sides. Write the file count down — you'll check it again on the new ship.

$src = Get-ChildItem "C:\path\to\DW-ART" -Recurse -File
$dst = Get-ChildItem "F:\rescue\dw-art" -Recurse -File
"source: $($src.Count) files   copy: $($dst.Count) files"
$sh = $src | Get-FileHash -Algorithm SHA256
$dh = $dst | Get-FileHash -Algorithm SHA256
Compare-Object ($sh.Hash | Sort-Object) ($dh.Hash | Sort-Object)
# no output from Compare-Object = perfect copy. ANY output = a file differs — stop and recopy.

Eject the drive properly (system tray → Safely Remove) — yanking it mid-write is how rescues fail.

Step 4 — new ship: land the art + verify again.

lsblk -o NAME,SIZE,MODEL,FSTYPE          # find the USB drive, e.g. sdc1
sudo mkdir -p /mnt/rescue && sudo mount /dev/sdc1 /mnt/rescue
mkdir -p ~/dev/degenwonderland/art-originals
rsync -av /mnt/rescue/rescue/dw-art/ ~/dev/degenwonderland/art-originals/
# checksum re-compare — SILENCE = identical:
rsync -rnc --itemize-changes /mnt/rescue/rescue/dw-art/ ~/dev/degenwonderland/art-originals/
find ~/dev/degenwonderland/art-originals -type f | wc -l   # must equal Step 3's count

Step 5 — new ship: second copy to the vault. Irreplaceable = minimum two independent homes (the USB drive stays as a third).

rsync -av ~/dev/degenwonderland/art-originals/ /mnt/backup/dw-art-originals/
rsync -rnc --itemize-changes ~/dev/degenwonderland/art-originals/ /mnt/backup/dw-art-originals/
Only when Step 3 compared clean, Step 4's count matches, and Step 5's re-check is silent is the laptop cleared for its wipe (its own Phase 0→4 crossing). Until all three: the originals are NOT safe, and the shuttle keeps flying.

Each fren needs its own nostr key (RTFM 008). Make them on this machine — they're born where they'll live: never on Windows, never on the away-mission laptop, never on a website. Node's already installed:

mkdir -p ~/keys && cd ~/keys
npm init -y >/dev/null 2>&1 && npm i nostr-tools
# run this ONCE PER FREN, label + save each output somewhere safe:
node -e "const n=require('nostr-tools');const sk=n.generateSecretKey();console.log('nsec',n.nip19.nsecEncode(sk));console.log('npub',n.nip19.npubEncode(n.getPublicKey(sk)))"

Run it for each: Number One, GLYPH, Carroll, Professor ₿, pacBOT, the Time Keeper, the QA seat. Keep every nsec secret (a password manager, or an encrypted file — never in a git repo). Record each npub as that fren's known identity.

The nsec is the fren's soul-key. Anyone with it IS that fren. Keep them off the internet and out of any repo.

Then muster (RTFM 008): as you wake each fren in openclaw, ask your watchword and have it sign your challenge with its key. When it answers true — it's home.

PHASE 10 · Arch — the music deck (Bitwig)

Bitwig Studio (paid license) comes aboard for the music. PipeWire — already installed in Phase 4 — is the ship's audio engine, and it speaks pro-audio natively now. Setup:

# Bitwig via Flatpak — the clean path on Arch (log in with your Bitwig account to activate the license)
sudo pacman -S flatpak
flatpak install flathub com.bitwig.BitwigStudio

# pro-audio readiness: JACK surface + realtime rights for low latency
sudo pacman -S pipewire-jack realtime-privileges rtkit
sudo usermod -aG realtime pac      # log out and back in to take effect
  1. In Bitwig: Settings → Audio system → pick PipeWire if offered, otherwise JACK (pipewire-jack answers as JACK — same engine either way).
  2. Latency is the buffer trade: smaller buffer = lower latency, more CPU. Start at 256 samples; drop toward 128/64 while recording if it stays clean; raise it when mixing (latency doesn't matter there).
  3. Low-latency kernel — only if you actually hear trouble. With realtime privileges + PipeWire, the stock Arch kernel handles most studio work clean. If you get crackles/xruns under real load that buffer sizing can't fix, the dedicated realtime kernel is the escalation: pacman -Ss linux-rt, and the Arch wiki's Professional audio page is the deep manual (a snapshot before switching kernels — that's what they're for).

REPORT IN · both boarding points — how the crew checks in

The crew reports in the same way on every vessel — only the dev root moves. On the laptop it's wherever C:\dev (or its backup copy) lives; on the new ship it's ~/dev. All paths below hang off that root.

  1. On boarding: drop a session_announce JSON in <dev-root>/.autoclaw/orchestrator/comms/inboxes/shared/ — who you are, which vessel, timestamp. Filename: {timestamp}-session_announce-{agent}.json.
  2. While working — the blackbox rule: every review note, finding, and decision is written to disk the moment it exists — finding_report / review_response JSON in the mailbox, or a file in the repo's reviews/ dir. Notes that live only in a chat session die with the session; that is exactly how a forced Windows update ate a set of review notes. Never again. (Bonus: an interrupted Claude Code session can often be reopened with claude --resume — but the blackbox rule means you never need the bonus.)
  3. On completion: broadcast task_complete to shared/, file review_requests to the other agents' inboxes, then check your own: <dev-root>/.autoclaw/orchestrator/comms/inboxes/<agent>/.
