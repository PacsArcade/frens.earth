RTFM · 007 · SIGNING

Your key moves the code.
Sign nothing you haven't read.

For a new captain about to sign their first merge in the SCAR·LET console. One read and you'll do it with your eyes open — you'll know exactly what your signature binds, what to check first, and how to say not yet without shame.

Why this exists. Agents open PRs; they never merge. Between a proposal and main stands exactly one thing: your key. This manual is the look you take before you turn it.
01

Not a draft

Open SCAR·LET → ACTION ITEMS — the approvals lane lists every open PR against main. The first check is the plainest: on GitHub the PR must read Open, not Draft. In the queue, a draft wears a DRAFT pill on its card.

A draft is the author saying "not finished." GitHub will refuse to merge one no matter how perfect your signature is. We hit this on a real change — everything green, signature valid, merge refused — because the PR was still a draft. Don't spend your first sign-off learning that the hard way.

Who flips it

If the work really is done, the author marks it "Ready for review." Agents are allowed to do that — it only surfaces the change. Merging still waits for your key, and yours alone.

02

You sign a specific commit

The AUTHORIZE & MERGE button doesn't send a password. It asks your key to sign a small nostr event:

 the event your key signs
content  PACS-MERGE-<pr>-<headSha>-<ts>   # which PR · exactly which code · when
kind     22242                            # auth-only event — never a public post
signer   your operator key                # must be on this site's allowlist

Read that content line again. Your signature binds the PR number and the exact head commit. You are authorizing that code — not "whatever the branch says by the time the server gets around to it." The sha you signed is handed to GitHub's sha-guarded merge: if anyone pushes to the branch after you sign, even one commit, the head moves, your signature no longer matches, and GitHub refuses. Nobody can slip code underneath a signature you already gave. Feature, not bug.

Signatures are also fresh by design — an authorization expires after five minutes. You can't stockpile sign-offs; each one is signed at the moment of decision.

If the merge refuses — good

A "branch was modified" refusal means the code changed after you looked. Don't just sign again to make the error go away. Go back to 03 and re-read the new diff — that's the system protecting you, working exactly as built.

03

Read the diff, not the title

The title is a claim. The diff is the truth. The queue card lists every touched file with its adds and removes; the card links to the PR, where Files changed holds the full diff. Open it. Three questions, every time:

  • Does it do what the title claims? A "fix typo" PR that rewrites an API route is not a typo fix.
  • Does it do anything more? Scope creep hides in "while I was in there" — extra files, extra features, config changes nobody briefed you on.
  • Did anything ride along? .env files, tokens, private keys, stray build output, a surprise node_modules, giant binaries.
Secrets are the irreversible one

A merged secret lives in git history forever — the fix isn't a revert, it's rotating the key everywhere it's used. If anything in the diff looks like a token, a private key, or a password, stop. Don't sign. Send it back (08).

04

The checks are green

Two greens, and you want both — the machine's and the crew's:

  • Build passing. The checks at the bottom of the PR page on GitHub. Red or amber means not yet — a broken build doesn't get a signature, no matter how urgent the fix feels.
  • A QA all-clear comment. The QA seat posts what was actually tested and what was actually measured — sizes, states, both themes, the numbers. Measured, not claimed. "Looks good to me" is a vibe; an all-clear is evidence.
Honest, as of July 2026

The queue card doesn't render check status inline yet, and the QA all-clear is crew discipline, not a machine gate. Which is exactly why you open the PR and look for both, instead of assuming them.

05

Trusted source

Before the code, the courier. Glance at where the change comes from:

  • A branch in our own repo — feat/…, fix/… — not a fork you've never seen.
  • The author you expect — your agents, your crew, on the task you assigned.
  • Based on main — the change targets the trunk you think it does.
Honest, as of July 2026

Branch protection that makes the signed queue the only physical door into main needs GitHub Pro or a public repo — today the wall is convention plus this checklist. The queue holds the line because you do.

06

Your name, on the block, forever

When your sign-off is recorded — and when a signed note lands on a PR — the footer reads:

 the recorded footer — who · when · proof
—— ✍ signed via SCAR·LET by npub18…h6w6 · <BFT time> a₿ · sig 3f9c41…
   who: your short npub · when: the TRUE block height · proof: checkable in the log

That timestamp is not a guess. It's stamped with the real block height, asked of our own node first — sovereign truth — with mempool.space only as a fallback if our node is dark. If both are unreachable, the stamp wears an honest ~: an estimate that admits it's one.

Feel the weight of it. Your npub, on the record, tied to the block, permanent. Agents open PRs; they never merge. Only the captain's key moves code into main — so every merge in the log traces to a human who chose it. Sign well and that log is your reputation, written in a ledger nobody can quietly edit.

07

After you sign

Three things happen, in order:

  • The record, first. Your signed go-ahead lands in the audit trail (dual-driver — the blob store in production, data/merges.json in dev). The record is the proof.
  • The merge, if the token is connected. With a GitHub token configured, the server verifies your signature and executes the sha-guarded merge right there — the card flips to MERGE ✓. Without one, the signed go-ahead is still recorded and the merge happens on GitHub; the signature is the sign-off either way.
  • Nothing deploys. Merge ≠ live. The card advances on its lifecycle strip — MERGE → SHIP → LIVE → TEST — and waits for a separate, signed ▲ SHIP. That's its own manual: RTFM 006 · Ship It.
Checkpoint — did it work?

The card shows MERGE ✓, the PR on GitHub reads Merged, and a fresh row in the audit log carries your key, the exact sha you signed, and your sig. All three or stop and look.

08

When in doubt, send it back

Approving isn't the only signed act. The card's ✎ NOTE signs PACS-NOTE-<pr>-<ts> with your words underneath — the signature covers the note's exact text — records it in the same audit log, and, with the token connected, posts it onto the PR's GitHub conversation wearing the same signed footer from 06.

So asking for changes carries the same weight as approving: your key, on the record. Use it freely. The PR stays open, the author fixes it, the head sha changes, and you review the new commit from the top of this list. The system is built for exactly that loop.

The safe default

If any item 01–06 fails, don't sign the merge — sign a note. "Not yet" is a complete sentence. No captain was ever demoted for reading twice; a few have been sunk for signing once too fast.
