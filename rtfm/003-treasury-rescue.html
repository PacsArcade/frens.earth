A

Locked out (the DR drill)

btcpay-admin.sh

The panel rejects you. No email reset arrives (no SMTP — common). You are not locked out of anything: the panel guards the web door, SSH guards the whole house.

Step 1 — board the box

 ssh — the master key you already hold
$ ssh you@your-vps            # the VPS, not the website
$ sudo su -
$ cd $BTCPAY_BASE_DIRECTORY/btcpayserver-docker/
$ ls *.sh                     # the deployment's toolbox
  • you should see: the helper scripts — btcpay-admin.sh, btcpay-restart.sh, bitcoin-cli.sh and friends
  • no scripts? you're in the wrong folder or a non-docker install — stop and check before improvising

Step 2 — reopen registration, make a rescue account

 the official reset — no database surgery
$ ./btcpay-admin.sh reset-server-policy   # reopens the Register button
$ ./btcpay-restart.sh                     # if Register doesn't appear
  • open the site in a browser → Register → create a fresh account (e.g. rescue@yourdomain)
  • you should see: a normal signed-in BTCPay session — but a powerless one, so far

Step 3 — promote it to server admin

 the promotion — run on the box
$ ./btcpay-admin.sh set-user-admin rescue@yourdomain
  • refresh the browser — you should see: Server Settings in the menu; you're an admin again
  • the new admin is not auto-added to any store — add yourself to the store from the server's user/store management
  • then lock the door behind you: Server Settings → Policies → disable public registration again
The gentler routes (when they exist)

If SMTP is configured (Server Settings → Email), Forgot password just… works — the reset arrives by mail and none of the above is needed. On older versions there's also the community Admin Pass Reset plugin (R0ckstar Dev). The SSH route above is the one that works with nothing configured.

The real key is the box

Anyone with SSH root on the VPS can do everything this phase did. Guard the SSH key like the treasury key — it IS a treasury key. And the mirror lesson: lose SSH and the panel lock becomes real, so keep your VPS provider's console access recoverable too.

The prevention kit — fit it now, while you're in

  • second admin — keep the rescue account, credentials stored separately from the first (different password manager entry, different human if you have a crew)
  • configure SMTP — Server Settings → Email; ten minutes now buys the forgot-password route forever
  • 2FA with recovery codes — if you enable two-factor or a U2F/FIDO2 hardware key on the admin account, save the recovery codes off the box, on paper; a lost key without codes means running this phase again
  • write the drill date down — this playbook was proven 0018.04.15 a₿; re-prove it when the deployment changes
Checkpoint

You're signed in as a server admin, registration is re-locked, a second admin exists, and you know why none of this ever scared you: the panel was never the perimeter — the box was, and you hold it.

diagnosis
B

"My sats aren't moving" (the ladder)

getblockchaininfo

Deposits confirmed days ago; the wallet says 0. Breathe: the sats exist on the chain — the chain is the ledger, your node is just a reader. A reader that stopped mid-book reports the balance as of the page it's on. Climb the ladder top-down; stop at the first rung that fails.

rung 1on-chain confirmed?▶ rung 2bitcoind synced?▶ rung 3lightning synced?▶ rung 4wallet rescan▶ rung 5restart order

Rung 1 — did the sats actually land? (ask the world, not your node)

  • open mempool.space → paste the deposit address or txid
  • you should see: the transaction with confirmations counting up — that's the world agreeing your sats exist
  • unconfirmed with a tiny fee? that's a mempool problem, not a node problem — wait, or fee-bump (RBF/CPFP) from the sending wallet
  • confirmed? good — the money is real; everything below is teaching your node to see it

Rung 2 — is bitcoind caught up?

 run on the box, in btcpayserver-docker/
$ ./bitcoin-cli.sh getblockchaininfo   # the node's own confession
$ ./bitcoin-cli.sh getblockcount       # one number, compare vs mempool.space
  • read three fields: "blocks" vs "headers", and "initialblockdownload"
  • blocks < headers → the node knows the chain is longer than what it has verified — it is behind; this is your whole problem
  • "initialblockdownload": true → it's still in first-sync mode; "verificationprogress" below 0.999… says how far
  • getblockcount equal to the tip on mempool.space → this rung passes; climb on
  • behind? jump to phase C to watch and unstick it — then come back

Rung 3 — is the lightning node caught up and connected?

 pick YOUR implementation — the deployment runs one of these
# Core Lightning (CLN):
$ ./bitcoin-lightning-cli.sh getinfo   # blockheight should match rung 2
# LND:
$ ./bitcoin-lncli.sh getinfo           # look for "synced_to_chain": true
  • you should see: a block height that matches bitcoind's — lightning reads the chain through your bitcoind, so it can never be ahead of rung 2
  • rung 2 behind → this rung is behind too, automatically; fix sync first, lightning follows
  • errors instead of JSON → read the lightning container's logs (phase C) before touching anything

Rung 4 — teach the BTCPay wallet what it missed

  • in the panel: Wallet → Settings → Rescan — scans the node's UTXO set (scantxoutset) for coins on your derivation scheme
  • server owners only (needs the full node) — that's you, captain
  • use it when the node IS synced but the wallet still misses deposits — classic causes: an imported xpub with history, or the 20-address gap limit
  • you should see: after the rescan, the balance and past transactions appear
Rescan law

A rescan can only find what the node has already verified. Rescanning an unsynced node finds nothing and proves nothing — sync first (rung 2), rescan last. Running the ladder bottom-up is how captains waste a day.

Rung 5 — the restart order (when a component is wedged, not behind)

 gentlest first — run in btcpayserver-docker/
$ ./btcpay-restart.sh          # the standard restart — fixes a surprising amount
$ ./btcpay-down.sh             # full stop…
$ ./btcpay-up.sh               # …clean start (the deeper reset)
  • restarting does not lose sync progress — the chain data stays on disk; the node resumes where it stopped
  • restart, then re-run rung 2 — blocks should be climbing again
The admiral's case, solved

5k sats: confirmed on mempool.space days before (rung 1 ✓). getblockchaininfo: blocks far behind headers (rung 2 ✗) — the node was days behind on the VPS. Nothing was lost, nothing was moving, because the reader was mid-book. Unstick sync (phase C), watch blocks catch headers, and the balance walks in on its own — no rescan even needed when the wallet was native.

runbook
C

The sync runguide (standing orders)

docker logs

Phase B finds the stuck part; this phase watches it, unsticks it, and keeps it from sticking again. This is the runbook — it stays useful after the emergency.

Know your containers (once)

 names vary by deployment generation — read YOURS off the list
$ docker ps                    # every running container, real names
  • you should see names like btcpayserver_bitcoind, btcpayserver_lnd_bitcoin or btcpayserver_clightning_bitcoin, generated_btcpayserver_1
  • older/newer generators prefix differently (generated_…_1 vs btcpayserver_…) — trust docker ps, not the manual

Watch the sync — the logs tell the truth

 the watch kit — substitute your names from docker ps
$ docker logs --tail 10 btcpayserver_bitcoind        # UpdateTip lines = blocks landing
$ docker logs --tail 40 btcpayserver_lnd_bitcoin     # or …_clightning_bitcoin
$ docker logs --tail 100 generated_btcpayserver_1    # BTCPay itself
$ ./bitcoin-cli.sh getblockchaininfo                 # blocks vs headers, on repeat
  • healthy sync looks boring: new UpdateTip/progress lines every few seconds, blocks climbing toward headers
  • silent logs + a frozen block count = wedged; keep reading

Disk space — the classic silent killer

 check this FIRST when sync stalls with quiet logs
$ df -h                        # a full disk stalls sync without one honest error
$ ./btcpay-clean.sh            # purge unused docker images — quick space back
# still tight? prune the chain (keeps ~1 year of blocks, ~100 GB):
$ export BTCPAYGEN_ADDITIONAL_FRAGMENTS="opt-save-storage"
$ . ./btcpay-setup.sh -i
  • you should see: usage on the data volume below ~90%; a VPS at 100% is the single most common "mysteriously stuck" cause
  • smaller prune variants exist: opt-save-storage-s (~50 GB) · -xs (~25 GB) · -xxs (~5 GB)
  • lifetime-VPS captains: your disk never grows — pruning is not an optimization, it's the plan

Starved resources — the other stall

 is the box working hard, or being held back?
$ docker stats                 # CPU pegged → box too small; near-idle → host throttling
$ free -h                      # swap in use during sync → needs more RAM (or add swap)

The restart sequence — smallest hammer first

 escalate in order; re-check getblockchaininfo between steps
$ docker restart btcpayserver_clightning_bitcoin   # ① lightning only (or …_lnd_bitcoin)
$ ./btcpay-restart.sh                              # ② the whole stack, standard
$ ./btcpay-down.sh
$ ./btcpay-up.sh                                   # ③ full stop / clean start

Expected times — so you don't panic at normal

  • fresh full sync: 1–5 days on a modest VPS — fast at first, slow at the end; that's the design, not a fault
  • a node days behind (the admiral's case): minutes to a few hours to catch up — watch blocks climb
  • lightning after bitcoind catches up: minutes — it reads from a local node
  • the chain is hundreds of GB unpruned — another reason pruning is the lifetime-VPS plan
Reindex — only when the logs say so

If docker logs on bitcoind literally says "Please restart with -reindex or -reindex-chainstate to recover", the data is corrupted. The docker remedy: ./btcpay-down.sh, then rm -rf /var/lib/docker/volumes/generated_bitcoin_datadir/_data/blocks and …/chainstate, then ./btcpay-up.sh — the node re-downloads and re-verifies (this is the 1–5 day clock again). Never do this to a node that's merely slow.

The monthly drill — ten minutes, run it
  • ① df -h — data volume under 90%?
  • ② ./bitcoin-cli.sh getblockchaininfo — blocks == headers, initialblockdownload: false?
  • ③ lightning getinfo — height matches bitcoind?
  • ④ sign into the panel with the second admin — proves phase A's kit is still loaded
  • ⑤ log the block height reached in the SCAR ops todo (docs/ops-todos.md) — the record is the drill
Checkpoint

You can say, with commands to prove it: where the node is, where the chain is, what's between them, and which hammer comes next. That's the whole runbook. A captain who runs the monthly drill never meets phase B by surprise again.
