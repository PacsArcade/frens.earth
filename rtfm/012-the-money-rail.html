RTFM · 012 · recover the treasury door, open the store

The Money Rail.
Your store, your node, your sats.

For the captain locked out of their BTCPay, and for every artist wiring a shelf to their own node. After this manual: you're back in, your site's store exists, and sats cross the counter without ever touching anyone else's hands.

Why this exists. The house provides the rails and never holds the money. That only stays true if YOU hold the keys to the rail — including the day you lose them. Getting locked out isn't failure; it's the drill. Run it calm.
01

The lockout drill — recover your BTCPay admin

Two doors back in. Try them in order.

Door one — the email reset (works only if SMTP was configured)

Login page → Forgot password. If the server has email set up, the reset lands in your inbox and you're done. Most self-hosted ships never configured SMTP — if the page says email isn't configured, that's door two.

Door two — the server-side reset (the real drill)

SSH into the box that runs BTCPay (for the arcade: the VPS). Then: register a fresh account in the web UI, and promote it from the server's own console.

 on the btcpay host, in the docker deployment dir
# 1. in the browser: Register a new account, e.g. captain-recovered@yourdomain
#    (if Register is hidden, run step 3 first, then come back)
$ ./btcpay-admin.sh set-user-admin captain-recovered@yourdomain
# 3. only if registration was disabled in Server Policies:
$ ./btcpay-admin.sh reset-server-policy
$ ./btcpay-restart.sh

The new admin isn't a member of the old stores yet — from the new admin account, add yourself to the existing store (Store → Settings → Users), or simply carry on with a fresh store below.

Careful

Whoever can SSH that box can mint BTCPay admins. This is why the server's SSH keys are treasury-grade (RTFM 001, RTFM 003) — the money rail's real lock is the box, not the login form.

Tip

While you're in: set SMTP (Server Settings → Email) so door one exists next time — and lock registration back off in Server Policies if step 3 opened it.

Checkpoint

You're signed into BTCPay as an admin. The drill is complete — log the block height in the shiplog.

02

Create the site's store — one store per ship

Sidebar → Create Store. Name it after the site (frens.earth). Never share one store between two sites — separate books, separate webhooks, separate blast radius. Connect the wallet under Wallets → Bitcoin (+ Lightning if the node has it): sats land in that wallet, nowhere else.

Copy the Store ID from Store → Settings → General.

03

Mint the key — invoices and nothing more

Avatar → Manage Account → API Keys → Generate Key. Pin it to the new store, check exactly two permissions:

 the only two boxes a website key ever gets
btcpay.store.cancreateinvoice
btcpay.store.canviewinvoices
Careful

A key with store-modify rights can re-point where the money goes. The site asks for invoices; give it invoices. Nothing more, ever.

Copy the key when it's shown — it shows once.

04

The webhook — how "paid" reaches the shelf

Store → Settings → Webhooks → Create Webhook. Payload URL (swap in your domain):

 payload url
https://frens.earth/api/store/webhook/btcpay

"Send all events" is fine — the site ignores the noise. Save, and copy the generated webhook secret: a separate credential from the API key. Without it the site can't tell a real "paid" from a forged one.

05

Hand the values to the ship + the order vault

Vercel dashboard → the site's project (check the project name in the header) → Settings → Environment Variables → Production:

 the four names
BTCPAY_URL              # e.g. https://btcpay.pacsarcade.org
BTCPAY_STORE_ID         # step 02
BTCPAY_API_KEY          # step 03
BTCPAY_WEBHOOK_SECRET   # step 04 — NOT the api key

The order vault: the public shelf is a shopfront; orders carry private things (who bought, where it ships). Those live in a private KV database — Storage tab → Create → Redis → Connect Project (prefix KV). No vault = the site refuses checkout on purpose rather than put a buyer's address anywhere public.

Careful

Never set env values through a PowerShell pipe — it writes BLANKS silently (the 0018.04 ghost). Dashboard, or a Linux printf pipe.

Env changed → redeploy (house rule 9). Deep runbook with every click: docs/btcpay-wiring.md in the template repo.

06

The maiden voyage

  • ① /store — the "rail not connected" banner is gone.
  • ② /a/store — ADD A WARE, small price in sats, GO LIVE.
  • ③ Buy it yourself; pay the lightning invoice.
  • ④ Watch the receipt flip PAID ✓ — that's the webhook doing its one job.
  • ⑤ Order book → MARK FULFILLED. The loop is closed.
Checkpoint

Sats crossed the counter from your own wallet to your own node through your own site. Nobody else touched them at any point. That's the whole thesis, proven with one coffee's worth of sats.
