01

Get the ship (clone)

 one folder, whole repo
$ git clone https://github.com/PacsArcade/frens.earth frens.hip
$ cd frens.hip
  • clone = download the repo, once
  • the folder is your frens.hip — Help ImProve; your copy of the ship
  • needs git installed, nothing else
  • private repo? sign in first: gh auth login
  • you should see: a new frens.hip folder full of the ship's files
  • pull = refresh later: git pull
02

First flight (run it)

 the whole install
$ npm install
$ npm run dev
  • first run writes your .env.local itself
  • open http://localhost:3000
  • claim a tag — forge keys in-browser
  • hit PLAY — the buddy works already
Checkpoint

You claimed a tag and signed in, locally. That's the whole machine, running on your desk.

03

Make it yours (two lines)

 .env.local — your space, your domain
NEXT_PUBLIC_SPACE_NAME=yourspace
NEXT_PUBLIC_NIP05_DOMAIN=yourspace.example
  • space name = the @tag you issue
  • domain = where the site lives
  • everything else already works
04

Go live (deploy)

 vercel — once to set up, one line forever after
$ npx vercel login
$ npx vercel link          # create the project
$ npx vercel blob create-store my-registry
$ npx vercel env add SEAT_SECRET production
$ npx vercel env add OPERATOR_NPUBS production
$ npx vercel deploy --prod  # ← the publish button
  • every vercel command asks questions — the defaults are almost always right
  • link creates the project; the name you give it becomes your Vercel project name
  • SEAT_SECRET = any long random string (it signs the login cookies)
  • OPERATOR_NPUBS = your npub — this is what opens /a for you
  • GITHUB_TOKEN comes later, in phase 06 — merges from the panel
  • you should see: a live https://…vercel.app address printed — your ship's first berth
The two laws of deploys

① New env vars do NOTHING until you deploy again. ② Merging does NOT deploy — npx vercel deploy --prod is the publish button.

05

The rhythm (every change after)

 your two commands, forever
$ gh pr merge N --merge      # accept proposal N
$ npx vercel deploy --prod   # publish the truth
  • proposals arrive as numbered PRs
  • read it, merge it, deploy it
  • or approve at SCARLET (frens.earth/a/scar) — needs phase 06 once
  • order rarely matters; overlaps get flagged
The finish line

A live site whose every change passed through your hands. The merge is the ceremony; the deploy is the broadcast.

06

Hook up approvals at SCARLET (once)

So the MERGE QUEUE at frens.earth/a/scar can list and merge proposals, it needs one key from GitHub. Two steps, five minutes, once.

Step A — make the key (on GitHub)

  • go to github.com → Settings → Developer settings → Fine-grained tokens → Generate
  • token name: frens-earth-scar-merges — just a label, max 40 characters, paste nothing else in that field
  • expiration: pick 90 days from the dropdown (when it expires, repeat this phase)
  • Resource owner: PacsArcade (the org, not your personal account)
  • repository access: Only select repositories → just PacsArcade/frens.earth
  • permissions: Contents + Pull requests — both Read and write; Metadata adds itself; nothing else
  • Generate token → copy it right away (GitHub shows it exactly once)

Step B — paste it (on the site — no Vercel needed)

  • open frens.earth/a/scar, signed in as operator
  • the queue shows CONNECT YOUR GITHUB — PASTE, SAVE, APPROVE
  • paste the key → ▶ CONNECT — done, forever
  • SCARLET runs the handshake in front of you — KEY RECEIVED ▶ REACHING GITHUB ▶ QUEUE ONLINE — and says so plainly when you did great
  • the key stores write-only; nobody can read it back, not even you
The Vercel back door (only if the box is unreachable)

Vercel project adminpacmans-projects / frens-earth → Settings → Environment Variables → add GITHUB_TOKEN = the key → then deploy again (law ①: env changes need a deploy).

Checkpoint

SCARLET lists every open proposal with ✍ AUTHORIZE & MERGE. Your signature — bound to the exact code — is the whole approval. No more GitHub tabs.

07

Change control — your ship, your rules

Everything above is a change-control system wearing ceremony clothes. Name the parts and you can tune them:

  • the proposal — every change arrives as a PR; nothing edits the master copy directly
  • the review — you read what changed (SCARLET shows the summary; the GitHub link shows every line)
  • the approval — your merge signature, bound to the exact code it approves
  • the record — every authorization lands in the audit log; the repo history is the ledger
  • the release — the deploy, kept separate on purpose (law ②: approving ≠ publishing)

Want a second approver?

  • OPERATOR_NPUBS takes a list — add a first officer's npub and they can sign too
  • two-signature merges for high-stakes changes are on the roadmap — the sha-bound signature format already supports counter-signing
  • each frens.hip writes its own rules: solo captains merge alone; crews add officers
