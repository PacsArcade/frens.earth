Doors and floors — the one idea

doors & floors

A floor is where the thing actually happens — the chat relay, the homeserver, the game engine. A door is the address a fren types. The whole art of wearing your own name is: you always own the door, even when someone else built the floor.

youyour brand▶ dnsthe door▶ the appthe gate▶ a serverthe floor▶ everythingthe fleet
  • door — the hostname frens see: chat.frens.earth
  • gate — the check that runs before the door opens (who's knocking?)
  • floor — the running service behind it (a relay, a homeserver, a node)
  • station — one room on a chat floor (orbee's word for a NIP-29 group)
  • mask — putting your name on a door whose floor you didn't write

Worked example throughout: orbee — github.com/imperviousinc/orbee, Apache-2.0, the nostr NIP-29 group-chat client both houses use as the floor chat. Everything said about orbee below was verified against its source on 0018.04.15 a₿; where something couldn't be verified, this manual says so instead of guessing.

dns + rewrites
A

The door wears your name

dns + rewrites

Shipped and live. Both houses put their own hostname in front of the same floor — with two different patterns, and the difference is the lesson.

The arcade's pattern — the straight redirect

chat.pacsarcade.org sends every path straight out to orbee's own app (from pacsarcade-org/next.config.ts, read 0018.04.15 a₿):

 pacsarcade-org · next.config.ts — redirects()
{
  source: "/:path*",
  has: [{ type: "host", value: "chat.pacsarcade.org" }],
  destination: "https://app.orbee.chat/:path*",
  permanent: false,   // temporary ON PURPOSE — the domain comes home
}                     // when the arcade's own floor ships
  • the fren types your name, then the browser moves — the URL bar ends up reading app.orbee.chat
  • the door is branded; the ride is orbee's — honest and fine as a stepping stone
  • permanent: false is deliberate: browsers don't cache it, so the door can come home later without stale redirects haunting you

frens.earth's pattern — the gated rewrite

chat.frens.earth never sends anyone anywhere until the gate has looked at them. The host points at THIS app, and the app answers (from next.config.ts + src/app/chat/route.ts on this branch):

 frens-earth · next.config.ts — rewrites(), not redirects()
/* beforeFiles: the chat host's root rewrites to /chat — the gate */
{ source: "/",       has: [{ type: "host", value: "chat.frens.earth" }], destination: "/chat" }
/* fallback: anything on the chat host that isn't a real route also
   lands at the gate — deep links from the direct-door era included */
{ source: "/:path*", has: [{ type: "host", value: "chat.frens.earth" }], destination: "/chat" }
  • the gate (src/app/chat/route.ts): a signed-in fren bounces 307 on to the configured floor; anyone else meets /login
  • the floor's URL comes from effectiveChatNode() — stored config → CHAT_NODE_URL env → the house default — and is never exposed raw in any UI
  • the loop guard: if no real floor is linked (the default points at the door domain itself), the gate says so honestly instead of chasing its own tail — house rule, honest empty states
  • /login and /api stay reachable on the chat host on purpose: root-only + fallback, or nobody could ever get through the gate there
Redirect vs rewrite — the whole difference

A redirect tells the browser "go over there" — the fren leaves your name behind. A rewrite answers on your own host — so a gate can run first, and your name stays in the bar until the moment YOU choose to bounce them. The arcade chose speed; frens.earth chose the gate.

The floor wears your name

fork & patch

Phase A masks a door. Phase B is owning the floor itself — running your own orbee, then deciding how much of its face becomes yours. First, what orbee actually is (verified against the repo, 0018.04.15 a₿):

  • a client, not a server — a SolidJS + Vite single-page app; the chat itself lives on NIP-29 relays, and handles come from the Spaces protocol faucet
  • Apache-2.0 — fork-friendly with attribution (more below)
  • app.orbee.chat is just this SPA served from a Cloudflare Worker; orbee.chat is a static landing page — two Workers, one repo (wrangler.jsonc)
Honest flag — there is no docker

The commission said "docker/build per its README" — verified: the repo has no Dockerfile and no docker-compose, and the README documents only npm install / npm run dev plus Cloudflare Worker deploys. Self-hosting orbee = building a static SPA and serving the files. That's lighter than docker, not heavier — but don't go hunting for a compose file that doesn't exist.

Run the app from your VPS

 the whole self-host — node 20+, then static files
$ git clone https://github.com/imperviousinc/orbee && cd orbee
$ npm install
$ npm run build:app          # vite build → dist-app/
$ sudo cp -r dist-app /var/www/chat-floor
# nginx: serve it as an SPA — every path falls back to the app shell
server {
  server_name chat-app.frens.earth;
  root /var/www/chat-floor;
  location / { try_files $uri /app.html; }
}
$ sudo certbot --nginx -d chat-app.frens.earth
  • SPA fallback is load-bearing: station links like /s/… aren't real files — they must land on the app shell (upstream's own Worker config sets single-page-application fallback for the same reason)
  • then point /a/chat at this host (phase C) and the gate bounces frens here
  • honest note: these steps were verified against the repo's build scripts and Worker config, not by running a build on the VPS — expect the usual first-run friction

The floor under the floor — your own relay

The app is furniture; the conversation lives on a NIP-29 relay. Orbee's default is hardcoded: STATIONS_RELAY_URL = "wss://stations.orbee.chat" (src/lib/nostr.ts). Fully wearing your own name means your own relay — the orbee repo doesn't ship one, so you bring a NIP-29 relay like relay29 (khatru29 / strfry29 / relayer29 flavors) and stand it up at stations.frens.earth.

Station naming follows the NIP-29 canonical form <host>'<id> — apostrophe-separated, one copy-pasteable token. Orbee's own community station is stations.orbee.chat'orbee; yours becomes:

 station naming — theirs → yours
stations.orbee.chat'orbee        # orbee's station on orbee's relay
stations.frens.earth'frens       # your station on YOUR relay
# in the app the door is  /s/stations.frens.earth'frens

Branding the floor — what's config, what's a fork

No fork needed — your station's name (verified): NIP-29 station metadata is runtime data. An admin edit (kind 9002) sets name, about, picture, open/closed, and the relay re-emits it as kind 39000 to every client. Orbee's station settings UI drives exactly this. Your station is fully yours — name, blurb, avatar — on ANYONE's orbee app, today.

Fork needed — the app's own face (verified: no theming config, no env hooks, no white-label switch anywhere in the repo). Everything below is hardcoded, so re-skinning the app means fork-and-patch:

  • name — "Orbee" lives in app.html (title, OG/Twitter meta)
  • logo/icons — public/ (favicon, PWA icons, og-image; scripts/gen-icons.mjs regenerates the set)
  • colors — CSS custom properties in src/index.css (accent is fuchsia #d946ef; swap the token block for your palette)
  • fonts — Google Fonts links in app.html (self-host yours instead)
  • default relay — STATIONS_RELAY_URL in src/lib/nostr.ts; profile relays in src/lib/relay-pool-worker.ts
  • featured stations — the FEATURED array in src/components/ExploreView.tsx (point the shelf at YOUR stations)
  • the handle faucet — FAUCET_BASE in src/lib/faucet.ts points at faucet.spacesprotocol.org; keep it (handles stay protocol-level, not orbee-level) or run your own

One genuinely hopeful signal upstream: the station-config event (kind 30078) is documented in-source as carrying "future theme / emoji / welcome bits" — per-station theming looks planned, but as of 0018.04.15 a₿ it holds only scope + pins. Watch that file before forking deeper than you need.

The Apache-2.0 fork route — clean and legal

Fork on GitHub → keep the LICENSE file → add a NOTICE line ("based on orbee by Impervious, Apache-2.0") → state your changes (the patch list above IS the changelog). Keep patches small and mechanical so upstream merges stay cheap — and offer the theming hooks upstream first; if they land, your fork shrinks to a config file. Homage, not redesign: orbee's bee stays orbee's (the arcade's own design guide says the same).

Checkpoint

A relay answering at stations.frens.earth, your station minted and named, the app built from your fork serving under your host — and phase A's gate pointing at it. The whole stack wears the house name.

One fleet, many doors

nodeconfig

The template story, and the reason phases A and B stay separate: every frens.hip ships with the gate built in and the floor left blank. A new ship doesn't inherit our chat — it points at its own.

01fork the ship▶ 02stand up a floor▶ 03/a/chat — point▶ 04save · test▶ 05your door opens
  • /a/chat (shipped): point · save · test your orbee door from the operator console — honest REACHABLE / UNREACHABLE states, OPEN THE CHAT ▸ goes through the gate, never the raw node
  • precedence: stored config (what you saved at /a/chat) → CHAT_NODE_URL env bootstrap → the house floor at chat.frens.earth — a fresh fork's chat door still opens somewhere honest
  • same rail as the Spaces and MUD nodes: the app never runs the floor, it only points at it — chatUrl is just another node link
  • so the fleet picture: one template, N ships, each with its own door (phase A), each free to run its own floor (phase B), all wired through the same gate

Matrix, the headline row — two names, two rules

The classrooms ride Matrix, and Matrix splits your name in two. Get this wrong and it's wrong forever:

  • Rule 1 — the server_name is a tattoo. Synapse's server_name "will appear at the end of usernames and room addresses created on your server" and — quoting the docs — "cannot be changed later so it is important to configure this correctly before you start Synapse." The arcade chose pacsarcade.org, so every fren's door reads @tag:pacsarcade.org and classes live at #btc101-run1:pacsarcade.org. Choose the apex, not the machine name.
  • Rule 2 — the machine can live anywhere. Well-known delegation: serve /.well-known/matrix/server → {"m.server": "matrix.frens.earth:443"} at the apex, and IDs keep wearing frens.earth while the daemon answers at matrix.frens.earth. Identity and infrastructure, decoupled on purpose.
  • The client re-brands by config, no fork. Element Web is static files + one config.json (documented options): brand (the app's name everywhere), branding.auth_header_logo_url / welcome_background_url / auth_footer_links, default_server_config (lock the homeserver so there's no server picker), setting_defaults.custom_themes (full color themes), permalink_prefix (share links point at YOUR deployment).
  • The live proof is already at sea: the arcade's Element cabinet at portal.pacsarcade.org wears "Pac's Arcade Chat", an Arcade CRT custom theme, its own logo and footer links, homeserver pre-locked to pacsarcade.org — all of it one config.json on a static deploy, zero forked code (verified against the shipped config, 0018.04.15 a₿).
